## プロジェクト目的

国内株式市場に特化したスクリーニングアプリケーション。
個人投資家が銘柄選定の判断を「早く・正確に・迷わず」行えるようにすることを目指す。
特に、決算発表・業績修正・適時開示などのイベント発生時に即座に反応できる情報基盤を提供する。

### 解決する課題
- 約3,900銘柄の中から投資対象を効率的に絞り込めない
- 複数の財務指標を横断的に比較するのに時間がかかる
- 決算・PTS・受注など複数の情報源を別々に確認する手間がある
- リアルタイム性のある情報（PTS価格・適時開示・出来高急変）に素早くアクセスできない
- 決算サプライズやPTS急騰落など、通常と異なる動きを見落としやすい
- バリュー投資に必要な財務健全性指標（自己資本比率・有利子負債比率等）が一画面で確認できない
- 保有銘柄の決算発表日を見落とし、対応が遅れるリスクがある
- 投資初心者はスクリーニング条件に何を入力すべきか分からず、ツールを活用できない
- PC前提の設計で、通勤中やスキマ時間にスマートフォンから快適に使えない

### 提供する価値
- **速度** — 全銘柄を一括取得し、複数条件で瞬時にフィルタリング（3,900銘柄でも200ms以内目標）。大量データ処理にはWeb Workerと仮想スクロールを活用
- **即時性** — PTS価格・決算速報・適時開示を低遅延で反映。ポーリング方式（5〜30秒間隔）を基本とし、将来的にはWebSocket/SSEでリアルタイム更新を実現
- **異常検知・アラート** — 決算サプライズ（コンセンサス比乖離率）、PTS急騰落（変動率±5%以上）、出来高急増（過去20日平均比2倍以上）を自動ハイライト。閾値はユーザー調整可能。重要イベント時はブラウザ通知で即座にアラート
- **正確性** — 複数データソース（J-Quants / Yahoo Finance / TDnet）からの信頼性あるデータ。各数値にデータ出典を明示し、精度に注意が必要な項目には注記を表示
- **一元化** — ダッシュボード型UI（ウィジェット配置式）により、スクリーニング結果・決算速報・PTS動向・受注情報・リアルタイム株価を単一画面で統合表示。サイドパネルで銘柄詳細を表示し、メイン画面から離れずに深掘り分析が可能
- **すぐ使える** — 「高配当株」「割安成長株」「大型安定株」など目的別プリセット条件をワンタップで適用可能。初めてでも迷わない
- **財務指標の充実** — PER・PBR・ROE・配当利回りに加え、自己資本比率・営業利益率・EPS・BPS・FCFを提供。業種平均との比較も可能
- **深掘り** — 個別銘柄画面で業績推移・セクター内比較・事業概要を確認可能。自前データで網羅できない情報は株探・IR BANK等へワンタップで連携
- **決算見落とし防止** — 決算発表カレンダー、通期予想に対する進捗率の自動計算、保有・監視銘柄の決算日アラート通知を提供
- **カスタマイズ** — お気に入り管理（保有/監視/売却検討のグループ分類、複数ウォッチリスト対応）、表示カラムの選択・並び替え、ソート順の永続化、更新頻度の調整、レイアウトプリセット保存、キーボードショートカット対応
- **どこでも使える** — モバイルファースト設計。スマートフォンでもPC版と同等の機能を快適に利用可能

### 対象ユーザー
- ファンダメンタルズ分析をベースに銘柄選定を行う個人投資家（特にバリュー投資家・高配当株投資家）
- 決算発表やPTS動向を素早くチェックしたいアクティブトレーダー
- 決算サプライズや業績修正をトリガーとしたイベントドリブン型トレーダー
- 投資を始めて間もなく、スクリーニングの使い方を学びながら活用したい初心者

### UI設計原則
- **専門用語には必ず補足を付ける** — PER・PBR・ROE等にツールチップで意味・一般的な目安値を表示
- **段階的開示** — 情報量の多い画面はデフォルトで主要項目のみ表示し、詳細はアコーディオン・タブで展開
- **モバイルファースト** — ナビゲーションはモバイル時ハンバーガーメニュー化。テーブルはカード型レイアウトに切替。タッチ操作に適したボタンサイズ（44px以上）を確保

### 利用シーン
- **通勤中（スマホ）** — お気に入り銘柄やプリセット条件でサッと確認。PTS値動きの速報チェック
- **帰宅後（PC）** — 詳細なスクリーニング条件設定、決算データの横断比較、CSV出力による分析
- **決算シーズン** — リアルタイムビューアで発表直後の業績を把握し、PTS動向と合わせて翌営業日の判断材料とする

### 実データ接続ロードマップ
現在モックデータで動作している機能について、段階的に実データソースへ接続する：
1. **Phase 1** — 決算データ（J-Quants fins/statements or 適時開示API）
2. **Phase 2** — PTSデータ（SBI証券 or ジャパンネクストPTS情報）
3. **Phase 3** — 受注データ（各社IR情報の構造化取得）
4. **Phase 4** — WebSocket/SSEによるリアルタイム配信基盤

## プロジェクト概要

国内株式スクリーニングWebアプリケーション（Next.js 14 App Router + TypeScript + Tailwind CSS）。複数のAPIからリアルタイム株価データを取得し、各種財務指標（PER、PBR、ROE、配当利回り、時価総額、出来高、株価）でフィルタリングが可能。

## アーキテクチャ

### データソースのフォールバック構成

`lib/api/jquants.ts` が以下の優先順位でデータ取得を制御：
1. **J-Quants V1**（`jquants-implementation.ts`）— メール/パスワード認証
2. **J-Quants V2**（`jquants-v2.ts`）— APIキー認証
3. **Yahoo Finance**（`yahoo-finance.ts`）— 無料、約3,900銘柄、crumb/cookie認証
4. **モックデータ** — 10銘柄のハードコードデータ（最終フォールバック）

### APIルート

- `GET /api/stocks` — キャッシュ済み株価データを返却（6時間TTL）。認証はリクエストヘッダー経由（`x-jquants-email`、`x-jquants-password`、`x-jquants-api-key`、`x-api-base`）。`?clearCache=true`でキャッシュバイパス。
- `POST /api/update` — キャッシュクリア後、最新データを再取得。

サーバー側キャッシュは `lib/api/cache.ts`（TTL付きインメモリシングルトン）を使用。

### クライアント状態管理

- **React Query**（`@tanstack/react-query`）— クエリキー `['stocks']`、staleTime 1分、ウィンドウフォーカス時の再取得無効。設定は `app/providers.tsx`。
- **localStorage** — 認証情報（`jquants_*`）、お気に入り（`stock-screening-favorites`）、表示件数設定を保存。
- 全ページコンポーネントは `'use client'` ディレクティブを使用。

### データフロー

```
クライアント (page.tsx) → fetchStocks()（認証ヘッダー付き）
  → GET /api/stocks（サーバー）
    → jquants.ts フォールバックチェーン → Stock[]
    → memoryCache（6時間TTL）
  → クライアント側フィルタリング (lib/utils/screening.ts)
  → StockTable（ソート、ページネーション、CSVエクスポート）
```

### 主要な型定義

- `Stock` — 基本データモデル（code, name, market, price, marketCap, volume, per, pbr, roe, dividendYield, updatedAt）
- `ScreeningCriteria` — 各指標のフィルタ範囲
- `StocksResponse` — APIレスポンスラッパー（stocks[], total, updatedAt）
- 財務関連型（`OrderData`、`PTSData`、`EarningsData`）は `lib/types/financial.ts` に定義

## 実装上の注意点

- Yahoo Finance APIは `lang=ja-JP&region=JP` パラメータがないと英語の銘柄名が返る。
- Yahoo Financeの配当利回りは小数（0.02 = 2%）で返るため、アプリ側で100倍して表示。
- Yahoo Finance使用時のROEはEPS/BookValueから算出。
- J-Quants V2は株価・出来高のみ提供（PER、PBR、ROE、配当利回りは取得不可）。
- 小型株のYahoo Finance時価総額データは信頼性が低い。
- Vercelデプロイリージョン：nrt1（東京）。設定は `vercel.json`。
- APIルートのCORSヘッダーは `Access-Control-Allow-Origin: *` に設定。
